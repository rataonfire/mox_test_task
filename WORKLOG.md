## CP-1 — Архитектурный план и первые ошибки AI (Phase 1)
Задача, которую я поставил AI (агент-архитектор, чистый контекст): составить PLAN.md по спецификации §2–§4 — дерево компонентов, модель состояния, замороженный API движка, 3 слайса сборки, таблица edge-cases, out-of-scope. Кода не писать.
Что предложил AI: полный PLAN.md с 8 флагами [DECISION], 17 edge-cases, 10 smoke-тестами.
Моё решение: принял план со **тремя исправлениями** (ошибки архитектора, найденные ручной сверкой со спецификацией):
1. **[DECISION #2] отклонён** — архитектор предлагал «реверс-инжинирить» константу uncertaintyWidth, хотя формула из ТЗ (`estimate × (1 ± (1 − confidence) × 0.5)`) даёт эталонные 7–11 точно: 9×0.81=7.29→7, 9×1.19=10.71→11.
2. **[DECISION #6] отклонён — критично** — архитектор прочитал FR-11 как runtime-лог пересчётов («Пользователь отредактировал сигнал»). По ТЗ MOX это журнал AI-процесса разработки (5–7 курируемых чекпоинтов: промпт → результат → решение → проверка). Runtime-лог означал бы автоматический незачёт.
3. **Шаг 3 движка исправлен** — архитектор округлял оценку по каждой локации до суммирования; по ТЗ суммируются неокруглённые значения, округляется только итог (на seed-данных совпадает случайно: 2+3+2+2=9, но в общем случае — ошибка).
4. **[DECISION #4] изменён** — вместо регистрозависимых локаций («Сарай» ≠ «сарай») выбрана нормализация (trim + без учёта регистра) для группировки: иначе опечатка фермера незаметно завышает оценку, разбивая одну территорию на две.
Проверка: ручной пересчёт эталонного примера из §7 на бумаге (8.94 → ≈9; 62%; 7–11); все правки внесены в PLAN.md с пометками [CORRECTED/AMENDED by orchestrator].

**🚦 GATE 1 пройден** — план представлен, возражений не поступило в отведённое окно; политика «молчание = согласие» была объявлена заранее. Все [DECISION] закрыты в пользу спецификации.

## CP-2 — Слайс 1: движок оценки + тесты (Phase 2, S1)
Задача, которую я поставил AI: workflow из трёх агентов — (1) имплементер: чистая функция `estimate()` + `recommendations.ts` + `validate.ts` + `defaults.ts` + полный Vitest-набор с закреплённым эталоном §7; (2) два параллельных адверсариальных проверяющих: один НЕЗАВИСИМО пересчитывает seed-математику из спецификации до чтения кода и сверяет с ожиданиями тестов, второй проверяет чистоту функции, замороженный API, генерацию объяснений из вычислений.
Что предложил AI: движок зелёный с 32 тестами (эталон: 8.94 → 9 кроликов, 62% «средняя», диапазон 7–11 — совпал с ручным пересчётом). Проверяющие нашли 4 недочёта (blocking — 0): (а) в причине рекомендации про ямки сумма `count` подписана как «сигналов» (для seed: 1 сигнал, но «сигналов: 2»); (б) захардкоженные '0' в объяснениях факторов вместо интерполяции; (в) сломанные русские множественные формы («2 локация»); (г) вес со слайдера мог утечь в объяснение как 0.30000000000000004.
Моё решение: все 4 исправил сам (точечные правки — делегировать 4 однострочника дороже, чем сделать): добавил `format.ts` (pluralRu + round2), переписал объяснения факторов на интерполяцию, «новых ямок: N» вместо «сигналов: N», `toFixed(1)` для средней интенсивности.
Проверка: `npm test` — 32/32 зелёные, `npm run build` — exit 0; независимый пересчёт математики верификатором сошёлся с тестовыми ожиданиями до внесения правок (правки не трогали числа — только строки объяснений).
